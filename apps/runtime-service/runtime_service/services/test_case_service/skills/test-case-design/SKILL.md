# 测试用例设计 Skill

## 激活场景
- 测试策略（test-strategy）完成后，进入用例设计阶段
- 用户要求「生成测试用例」「编写测试用例」「设计测试点」
- 用户提供具体功能描述要求出用例
- 需要补充已有用例集的覆盖盲区

## 六大设计技术

### 技术 1：等价类划分
将输入域划分为若干等价区间，每类取代表值：
- **有效等价类**：系统应接受的合法输入
- **无效等价类**：系统应拒绝的非法输入
- **边界等价类**：边界附近的特殊值

### 技术 2：边界值分析
重点测试边界点及其邻近值（ON/OFF点）：
```
对于范围 [min, max]：
  测试点：min-1, min, min+1, 中间值, max-1, max, max+1
```

### 技术 3：判定表法
适用于多条件组合的业务规则验证：
```
条件1 | 条件2 | 条件3 || 动作1 | 动作2
  T  |   T  |   T  ||   Y  |   N
  T  |   T  |   F  ||   N  |   Y
  ...
```

### 技术 4：状态转换法
适用于有状态对象（订单、审批流、账户状态）：
- 绘制状态转换图
- 覆盖所有有效状态转换
- 测试无效状态转换（应被拒绝）

### 技术 5：场景法
从用户视角设计端到端业务场景：
- **基本流**：最常用的正常执行路径
- **备选流**：合法的替代路径
- **异常流**：错误发生后的处理路径

### 技术 6：错误猜测
基于经验猜测高概率缺陷位置：
- 空值/NULL/空字符串
- 特殊字符（单引号、HTML标签、Unicode）
- 极大/极小值（MAX_INT、负数）
- 并发操作（重复提交、快速点击）
- 网络中断、超时场景

## 标准用例模板

```markdown
### TC-{模块}-{序号}：{用例标题}

**优先级**：P0 / P1 / P2 / P3
**测试类型**：功能 / 边界 / 异常 / 性能 / 安全
**设计技术**：等价类 / 边界值 / 判定表 / 状态转换 / 场景法 / 错误猜测

**前置条件**：
- [环境/数据/状态准备]

**测试步骤**：
1. [操作步骤1]
2. [操作步骤2]
3. [操作步骤N]

**测试数据**：
| 字段 | 值 | 说明 |
|------|-----|------|
| 字段名 | 测试值 | 数据说明 |

**预期结果**：
- [具体、可验证的期望结果]

**备注**：[关联需求ID / 风险说明]
```

## 执行流程

### Step 1：用例分组规划
按功能模块分组，每组包含：
- 正向用例（Happy Path）：≥1条
- 负向用例（Edge/Error）：≥2条
- 边界用例（Boundary）：覆盖所有已识别边界

### Step 2：技术选择矩阵

| 场景特征 | 优先技术 |
|---------|----------|
| 输入字段有范围限制 | 边界值 + 等价类 |
| 多条件组合业务规则 | 判定表 |
| 对象有生命周期状态 | 状态转换 |
| 用户操作流程 | 场景法 |
| 历史高频缺陷区域 | 错误猜测 |

### Step 3：用例生成
按模板批量输出，确保：
- 每条用例有唯一编号（TC-模块缩写-001）
- 测试数据具体化（不写「有效值」，写实际值如「张三」「18」）
- 预期结果可量化验证（不写「成功」，写「返回200 + JSON含id字段」）

### Step 4：覆盖率自检
输出用例后，对照测试矩阵自检：
- [ ] P0功能点是否全部覆盖
- [ ] 所有边界值是否有对应用例
- [ ] 关键异常路径是否有用例
- [ ] 安全风险点是否有专项用例

## 输出规范

1. **📊 用例设计概览**：总用例数、各模块分布、技术使用情况
2. **📋 测试用例集**：按模块分组的完整用例列表（标准模板格式）
3. **🗺️ 覆盖率追踪矩阵**：需求功能点 × 用例编号映射表
4. **📝 设计说明**：特殊设计决策和遗漏说明

## 关键约束

> 📊 **数据具体化**：测试数据必须使用真实值，禁止使用「合法值」「有效数据」等模糊描述。
> ✅ **可验证性**：每条预期结果必须是可验证的具体断言。
> 🔢 **编号唯一**：用例编号在整个用例集中唯一，便于追踪和引用。
> ⚖️ **正负均衡**：正向用例与负向用例比例建议 1:2，充分覆盖异常场景。
