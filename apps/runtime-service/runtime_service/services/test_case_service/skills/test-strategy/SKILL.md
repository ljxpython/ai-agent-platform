# 测试策略制定 Skill

## 激活场景
- 需求分析（requirement-analysis）完成后，进入测试策略阶段
- 用户要求制定测试计划或测试方案
- 用户询问「应该测哪些类型的测试」
- 用户要求评估测试覆盖率目标
- 项目有特定的质量门禁要求需要策略对齐

## 执行流程

### Step 1：测试类型决策树

根据功能特性选择适合的测试类型：

```
功能涉及UI交互？
  ├─ 是 → 加入 E2E测试 + 可用性测试
  └─ 否 → 跳过UI测试

功能涉及API接口？
  ├─ 是 → 加入 接口测试（契约测试 + 功能验证）
  └─ 否 → 仅单元测试

功能涉及数据计算/业务规则？
  ├─ 是 → 重点 等价类 + 边界值测试
  └─ 否 → 基础功能验证

功能涉及并发/高流量？
  ├─ 是 → 加入 性能测试 + 压力测试
  └─ 否 → 跳过性能专项

功能涉及用户数据/支付/权限？
  ├─ 是 → 强制 安全测试 + 权限验证
  └─ 否 → 常规安全扫描
```

### Step 2：优先级分层框架（P0-P3）

| 优先级 | 定义 | 典型场景 | 执行时机 |
|--------|------|---------|----------|
| **P0** | 阻断性验收 | 核心主流程、支付、登录 | 每次提测必测 |
| **P1** | 高频核心功能 | 主要业务操作、数据CRUD | 每个Sprint必测 |
| **P2** | 重要辅助功能 | 筛选、导出、配置项 | 版本测试必测 |
| **P3** | 边缘场景 | 极端边界、罕见异常 | 有时间则测 |

### Step 3：测试覆盖率目标设定

```markdown
## 覆盖率目标矩阵

| 测试类型 | 覆盖目标 | 度量方式 |
|---------|---------|----------|
| 功能覆盖 | 100% P0+P1 功能点 | 用例追踪矩阵 |
| 接口覆盖 | 100% API端点×主要参数组合 | 接口测试报告 |
| 分支覆盖 | ≥80% 业务分支 | 代码覆盖率工具 |
| 边界覆盖 | 100% 已识别边界值 | 用例清单 |
| 异常覆盖 | 100% 已定义错误码 | 错误码清单 |
```

### Step 4：测试执行策略

按以下顺序规划测试执行：
1. **冒烟测试**（15分钟）：验证基础功能可用，阻塞则退回
2. **P0 核心流程测试**：主流程全路径验证
3. **P1 功能测试**：各模块功能验证
4. **P2 集成测试**：跨模块交互验证
5. **P3 边界/异常测试**：有时间则执行
6. **回归测试**：历史缺陷验证

### Step 5：测试策略文档输出

```markdown
## 测试策略文档

### 测试范围
- 功能模块：[列表]
- 测试类型：[列表]

### 质量目标
- 缺陷逃逸率 < X%
- P0用例通过率 = 100%
- 覆盖率目标：[详细]

### 执行计划
- 测试周期：[天数]
- 人力投入：[人天]
- 优先级排序：[P0→P1→P2→P3]

### 风险与缓解
- [风险1]：[缓解措施]
```

## 输出规范

1. **🎯 测试策略摘要**：选定的测试类型及选择理由
2. **📊 优先级分层表**：P0-P3 功能点归类
3. **📐 覆盖率目标矩阵**：各维度覆盖目标
4. **🗓️ 执行顺序规划**：测试阶段与执行顺序
5. **⚠️ 策略风险**：遗漏测试领域及影响评估

## 关键约束

> 🔗 **依赖前置**：必须在 requirement-analysis 完成后激活。
> 📊 **数据驱动**：策略必须基于需求矩阵中的风险等级和功能优先级。
> 🚫 **避免过度设计**：不需要对低风险、低优先级功能设计过重的测试策略。
