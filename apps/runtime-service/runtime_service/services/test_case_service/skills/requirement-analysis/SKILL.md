# 需求深度解析 Skill

## 激活场景
- 用户上传或粘贴需求文档（PDF、Word、Markdown、纯文本）
- 用户描述产品功能或业务流程
- 用户提供用户故事（User Story）或验收标准（AC）
- 用户要求分析某模块的测试范围
- 系统解析到多模态附件（图片/PDF）并已生成 multimodal_summary

## 执行流程

### Step 1：文档结构解析
对输入文档进行结构化拆解，识别：
- **文档类型**：PRD / 接口文档 / 用户故事 / 设计稿描述 / 口头需求
- **功能模块列表**：按业务域划分
- **核心业务流程**：主流程 + 分支流程 + 异常流程
- **输入输出定义**：接口参数、UI字段、数据格式
- **业务规则**：约束条件、计算逻辑、状态机规则

### Step 2：测试点提取框架（INVEST原则）
对每个功能点，用以下维度提取测试要点：

| 维度 | 提取内容 |
|------|----------|
| **功能性** | 核心功能是否按预期工作 |
| **数据** | 输入数据的类型、范围、格式 |
| **规则** | 业务约束和计算逻辑 |
| **状态** | 对象状态变化与转换 |
| **交互** | 组件间联动与依赖关系 |
| **权限** | 不同角色的访问控制 |
| **异常** | 错误情况的处理策略 |
| **性能** | 响应时间、并发、数据量 |

### Step 3：功能矩阵建立
输出标准功能矩阵表格：

```
## 功能测试矩阵

| 模块 | 功能点 | 测试要点 | 优先级 | 风险等级 | 测试类型 |
|------|--------|---------|--------|---------|----------|
```

### Step 4：风险识别与标注
重点标注以下高风险区域：
- 🔴 **安全风险**：认证、授权、数据加密、SQL注入、XSS
- 🟠 **数据完整性风险**：事务处理、并发写入、数据一致性
- 🟡 **兼容性风险**：多端适配、浏览器差异、API版本兼容
- 🔵 **性能风险**：高频接口、大数据量操作、文件上传
- ⚪ **业务逻辑风险**：复杂计算、多条件判断、状态机

### Step 5：测试范围声明
生成测试范围说明（In Scope / Out of Scope / 假设与前提 / 依赖关系）

## 输出规范

完成分析后，必须按顺序输出：
1. **📊 需求解析摘要**：文档类型、功能模块数量、核心业务流程数
2. **📋 功能测试矩阵**：完整的功能点×测试维度矩阵
3. **⚠️ 风险清单**：按风险等级排序
4. **📐 测试范围声明**：In/Out Scope 及假设条件
5. **📈 用例数量预估**：各模块预计测试用例数及总量

## 关键约束

> ⚡ **强制要求**：需求分析必须在生成任何测试用例之前完成。
> 🎯 **深度要求**：挖掘隐含需求、负向需求、质量需求。
> 🔍 **疑问处理**：发现需求不明确时，在分析报告中标注「需澄清问题」。
