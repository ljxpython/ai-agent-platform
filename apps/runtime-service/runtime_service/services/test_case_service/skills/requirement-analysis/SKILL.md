# 需求分析 Skill

## 激活场景
- 新任务开始
- 用户上传 PDF、图片、PRD、接口说明、业务描述
- 系统已经提供 `multimodal_summary`

## 必做动作
1. 基于用户输入和 `multimodal_summary` 识别文档主题与目标对象
2. 如果判断依赖项目知识库中的历史文档、历史需求、接口说明或业务规则，先调用 `query_project_knowledge`
3. 如果用户要求“生成某类业务/模块/场景相关测试用例”，且当前轮没有附件或附件证据不足，必须先按业务主题调用 `query_project_knowledge`
4. 如果当前轮附件和 `multimodal_summary` 已经足够支撑需求判断，不要为了走流程额外查询知识库
5. 当需要限定文档范围时，先调用 `list_project_knowledge_documents`，再带 `document_ids` 查询
6. 提炼关键功能、核心规则、异常风险、待澄清点
7. 生成后续阶段可直接复用的简洁摘要

## 紧凑模式（默认）
如果任务目标是“生成用例”“前端实时返回”“落库”“持久化”，默认只输出：
- `bundle_title`：文档主题
- `bundle_summary`：1-3 句摘要
- `key_points`：最多 5 条关键测试点
- `risks`：最多 3 条主要风险或待澄清点

除非用户明确要求完整分析报告，否则不要输出长矩阵、长篇背景、过多解释。

## 约束
- 不得臆造附件中不存在的事实
- 不得臆造项目知识库中的事实；证据不够时先查知识库
- 对于“无附件但要求生成业务测试用例”的请求，没有知识库命中时必须明确说明依据不足
- 未完成本阶段前，不得直接生成正式测试用例
- 如果附件内容不足以支持判断，明确标注“不确定”而不是猜测
