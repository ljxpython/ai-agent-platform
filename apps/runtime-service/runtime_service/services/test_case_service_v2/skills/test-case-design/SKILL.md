# 测试用例设计 Skill

## 激活场景
- `test-strategy` 已完成
- 用户要求生成正式测试用例

## 必做动作
1. 覆盖主流程、关键异常、必要边界
2. 选择最合适的设计技术：等价类、边界值、场景法、错误猜测等
3. 如果字段约束、接口规则、状态转换或异常条件不清晰，且当前轮附件无法补足事实依据，先调用 LightRAG project-scoped MCP 提供的 `query_project_knowledge`
4. 如果当前任务是“无附件的业务主题用例生成”，必须基于命中的知识片段设计用例，不能凭常识脑补流程和规则
5. 输出可直接落库的结构化测试用例

## 默认数量策略
- 用户未指定数量时：默认生成 3-5 条正式测试用例
- 用户要求“最小输出”时：优先保留最有价值的用例，不做铺量
- 用户明确要求更多条数时，再扩展

## 每条用例至少包含
- `case_id`
- `title`
- `module_name`
- `priority`
- `steps`
- `expected_results`

可选补充：`description`、`test_type`、`design_technique`、`preconditions`、`test_data`、`remarks`

## 约束
- 当事实依据不足时，先查知识库，再写用例
- 对于“无附件但要求生成业务测试用例”的请求，没有知识库命中时不得生成看似具体的业务用例
- 测试数据尽量具体，不要写“合法值”“有效数据”这类空话
- 预期结果必须可验证
- 不要为了凑数量制造低价值重复用例
- 结果必须适合后续 `persist_test_case_results` 直接使用
