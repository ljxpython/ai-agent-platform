# 测试策略 Skill

## 激活场景
- `requirement-analysis` 已完成
- 需要决定测试重点、优先级和覆盖方向

## 必做动作
1. 根据需求摘要识别必须覆盖的核心流程
2. 确定优先级：P0 核心主流程，P1 高频功能，P2/P3 补充场景
3. 如果需求摘要对业务规则、接口约束或边界条件说明不足，且当前轮附件不能补足证据，先调用 LightRAG project-scoped MCP 提供的 `query_project_knowledge` 补证据
4. 如果当前任务是“无附件的业务主题用例生成”，策略必须明确引用知识库检索到的业务片段，不能脱离检索结果空谈
5. 为用例设计给出最少必要的策略约束

## 紧凑模式（默认）
默认只输出：
- `strategy_summary`：1 段简述
- `priority_focus`：最多 3 条优先级策略
- `coverage_targets`：最多 3 条覆盖要求
- `execution_order`：一句执行顺序

除非用户明确要求完整测试计划，否则不要展开完整覆盖率矩阵、工期估算或大段方案说明。

## 约束
- 必须基于本轮需求摘要，不能脱离输入空谈
- 需要引用项目文档事实时，先查知识库，不要凭经验硬编
- 在“无附件 + 业务主题生成”场景下，没有知识检索依据就不要继续扩展策略细节
- 低风险内容不要过度设计
- 策略必须服务于后续正式用例生成，而不是单独写成长文档
