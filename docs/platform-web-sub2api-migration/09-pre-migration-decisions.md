# 09. Phase 1 前置决策收敛

## 目标

这份文档专门收掉 Phase 1 开工前剩下的几个关键决策：

1. `vue-i18n` 到底保留到什么程度
2. 品牌命名和主题命名怎么定
3. 演示环境到底按什么接

这些问题如果不先定，后面一进入新 app 初始化，马上就会有人开始随手发挥。

## 一、`vue-i18n` 策略已确认

### 结论

正式新 app：

- 保留 `vue-i18n` 能力
- 第一阶段只维护 `zh-CN`
- 不在 Phase 1 同时维护多语言内容

### 为什么这么定

原因很简单：

- 未来平台前端仍然需要国际化扩展能力
- 但当前迁移目标是尽快稳定落地，而不是把文案维护成本翻倍
- 先保留能力，再缩小首期维护面，是最稳的做法

### 具体执行规则

Phase 1 约束如下：

- 保留 `src/i18n/` 目录
- 保留统一的文案 key 访问方式
- 只提供 `zh-CN` 语言包
- 页面文案不直接硬编码在组件结构里

建议结构：

```text
src/i18n/
  index.ts
  locales/
    zh-CN.ts
```

### 当前不做的事

Phase 1 不做：

- 英文文案全量补齐
- 多语言切换完整产品化
- 翻译平台接入

但这里要注意一个边界：

- “不做全量英文落地”
- 不等于“前端里完全不保留语言切换产品位”

如果后续要对齐 `sub2api-base` 的成熟顶栏体验，那么：

- locale switcher 的交互母版值得保留
- 只是它在 Phase 1 不要求全站英文内容立刻补齐

## 二、品牌命名已确认

### 1. 工程命名

工程命名已经确认：

- 目录：`apps/platform-web-vue`
- 包名：`platform-web-vue`

### 2. 前端展示命名

前端展示层建议固定为：

- 英文主名称：`Platform Workspace`
- 中文描述名：`平台工作台`

这样定的原因：

- 与当前已有的 workspace 语义一致
- 比 `platform-web-vue` 这种工程名更适合出现在界面上
- 后续如果老板要求品牌替换，也只需要换展示层配置

### 3. 品牌命名边界

后续页面里应该区分两种名字：

- 工程名：只在代码和目录里出现
- 展示名：只在界面和文案里出现

明确要求：

- 不要把 `platform-web-vue` 这种工程名直接展示给用户

## 三、主题命名已确认

### 1. 主题系统分两层

后续主题系统建议拆成两层：

1. `color mode`
   - `light`
   - `dark`
2. `theme preset`
   - `workspace`

### 2. Phase 1 主题结论

Phase 1 先明确：

- 默认 `color mode = light`
- 默认 `theme preset = workspace`
- 保留暗色模式能力，但不把“多套视觉风格预设”作为 Phase 1 必须项

### 3. 为什么这样定

原因：

- 你前面明确要保留“可切换主题”的能力
- 但这次迁移的第一优先级仍然是把平台功能和成熟视觉一起稳定下来
- 如果一上来把多套风格预设一起做，风险和改动面会明显放大

结论：

- Phase 1 先把“整体跟着切”的主题能力做对
- 风格预设系统保留扩展位，但不抢关键路径

### 4. 命名约束

代码层建议固定：

- `theme.mode = light | dark`
- `theme.preset = workspace`

不要出现这种乱命名：

- `blueTheme`
- `bossTheme`
- `themeA`
- `themeB`

这种命名一多，后面自己都记不住。

## 四、演示环境接入方式已确认

### 1. 当前演示基线

当前可用演示基线已经明确：

- 平台后端：`http://localhost:2024`
- LangGraph 开发服务：`http://localhost:8123`
- 登录方式：用户名密码登录

### 2. 当前演示账号

演示账号按下面两组固定：

- 管理员：`admin / admin123456`
- 普通用户：`test / test`

如果后续环境里没有，就按这个约定补齐。

### 3. 演示优先级

汇报优先级继续保持：

1. agent 相关能力最重要
2. 其次是平台基础管理能力

### 4. 新 app 的接入原则

`platform-web-vue` Phase 1 接入规则：

- 平台接口默认对接 `http://localhost:2024`
- agent 相关调试允许识别 `http://localhost:8123`
- 但业务主链路仍然优先通过平台上下文和平台接口组织

这意味着：

- 登录、项目、用户、审计、testcase 等都以 `2024` 为主
- `8123` 不是展示给普通用户的固定配置入口
- 它更偏向开发调试和底层接线能力

## 五、当前仍未关闭的唯一前置项

到现在为止，Phase 1 真正还没关闭的前置项只剩一个：

- 去掉 `apps/platform-web-sub2api-base` 的嵌套 `.git` 边界

这个事情必须做，但它是代码与仓库整理动作，不是架构方向问题。

所以当前结论已经足够支撑进入 Phase 1 代码初始化。

## 六、最终结论

Phase 1 前置决策现在已经收敛成下面这套：

- 新 app 名称：`apps/platform-web-vue`
- 国际化策略：保留 `vue-i18n`，首期只维护 `zh-CN`
- 展示品牌：`Platform Workspace / 平台工作台`
- 主题体系：`light|dark + workspace preset`
- 演示环境：`2024` 平台后端、`8123` LangGraph、用户名密码登录

这批东西定下来后，Phase 1 就不该再反复讨论方向，而应该直接进入初始化和实现。
