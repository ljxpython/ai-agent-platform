# 12. Dev 环境 Sub2api 模板展厅

## 目标

`platform-web-vue` 的开发环境里，`UI 资产库` 不再只是资源目录。

现在的目标是更直接的：

- 把 `platform-web-sub2api-base` 的前端内容按模板方式整库展示
- 让后续开发者直接挑模板、看结构、抄源码块
- 不再要求开发者自己去翻上游项目源码树

## 当前形态

当前入口已经拆成 `Resources` 多页面结构：

- 汇总页：`/workspace/resources`
- 开发范式页：`/workspace/resources/playbook`
- 团队推荐页：`/workspace/resources/top-picks`
- 汇总页内含三类模板的精选预览，不再只是纯导航
- 页面模板：`/workspace/resources/pages`
- 组件模板：`/workspace/resources/components`
- 工程模板：`/workspace/resources/engineering`
- 兼容旧路由：`/workspace/ui-assets` 会重定向到 `/workspace/resources`

当前结构不是单页大杂烩，而是六页：

1. `资源总览`
2. `前端开发范式`
3. `团队推荐 Top 10`
4. `页面模板`
5. `组件模板`
6. `工程模板`

其中：

- 页面模板：`views/**/*.vue`
- 组件模板：`components/**/*.{vue,ts}`
- 工程模板：`api / composables / stores / router / i18n / styles / utils / types / main.ts / App.vue / views/**/*.ts`

测试文件和 `*.d.ts` 已排除，不参与模板展厅展示。

## 为什么这样改

之前的实现更像“文件索引”：

- 会告诉你源码在哪
- 会告诉你哪些页值得参考
- 但开发者看到的仍然是一堆路径和目录

这不符合“模板库”的目标。

同时，模板库在自动全量收录之后，又出现了第二个问题：

- 模板总量很大
- 同类模板很多
- 开发者虽然知道源码在哪，但依然不知道“默认该先选谁”

所以现在资源页不仅要解决“整库展示”，还要解决“可决策”。

现在改造后的要求是：

- 每个模板都要有独立卡片
- 最常用的 10 个模板要额外钉死为 `团队推荐 Top 10`
- 推荐模板必须有明确的 `首选模板 / 团队推荐` 标记
- 卡片先告诉你这个模板适合借什么
- 点开后继续看源码块和结构拆解
- 页面、组件、工程支撑都要一起收进来

也就是说，开发者进入 `Resources` 后，不应先看到一坨路径列表，而应该先进入汇总页，再按需要进入页面模板、组件模板或工程模板页。

现在这件事又进一步收敛了一层。

开发者进入 `Resources` 后，第一步不应该直接盲挑模板，而应该先进入：

- `前端开发范式`

这页专门回答：

- 新页面该选什么母版
- 哪些组件和交互必须统一
- `sub2api-base` 的参考能力该怎么借
- 进入模板库前应该按什么顺序做判断

同时，汇总页也不能空得像目录页。

所以现在的要求补充为：

- 汇总页负责统计、导航和精选模板预览
- 子页负责完整模板库查阅和源码拆解

## 数据来源方式

当前不是手工维护一大坨静态对象。

而是直接从上游源码目录自动收录：

- 数据文件：`apps/platform-web-vue/src/modules/examples/ui-assets-catalog.ts`
- 参考源码快照：`apps/platform-web-vue/examples/sub2api-reference/src`
- 通过 `import.meta.glob` 自动枚举模板文件
- 页面只消费分组后的模板元数据
- 模板详情按需加载源码内容，而不是把说明手工写死

这里已经不再直接依赖本地 `apps/platform-web-sub2api-base` 那个上游仓目录。

原因：

- 原目录体积大
- 带嵌套 `.git`
- 不适合直接作为正式仓库依赖

所以现在 dev 资源库依赖的是一份轻量参考快照，而不是整个上游仓。

这样做的好处：

- 上游新增模板后，这里天然可扩展
- 不会把模板库维护成过期台账
- 页面、组件、工程模板能统一收口

## 展示策略

Resources 模板展厅当前遵循下面的展示策略：

### -1. 先看开发范式，再看模板

Resources 当前新增一页：

- `/workspace/resources/playbook`

这页承担统一前端开发口径的职责。

默认顺序现在固定为：

1. 先看 `开发范式`
2. 再看 `团队推荐 Top 10`
3. 再进入页面模板、组件模板、工程模板

这样做的原因是：

- 模板告诉开发者“可以借谁”
- 范式告诉开发者“应该怎么借”

### 0. 先做三层，而不是直接扔出整库

每个模板子页现在都统一为三层：

- `推荐模板`
- `场景模板`
- `模板库`

含义分别是：

- 推荐模板：默认首选集合，解决“开发时先看谁”
- 场景模板：按列表页、看板页、认证页、弹窗、壳层、工程接线这些真实任务收口
- 模板库：把推荐模板和场景模板去重后形成完整可搜索集合，用来做横向比较

这个结构的核心不是减少模板源码，而是减少默认暴露给开发者的噪音，同时避免同一类模板被重复展示。

### 0.5 再加一层跨分类固定锚点

除了每个子页自己的 `推荐模板` 外，总览页现在还额外固定一组 `团队推荐 Top 10`。

这层和普通推荐模板的区别是：

- 普通推荐模板：解决“当前分类里先看谁”
- 团队推荐 Top 10：解决“整个资源库里最先该吃透哪 10 个模板”

这批模板不是平均给每个目录分配名额，而是按后续开发最常复用的壳层、列表页、分页、弹窗和表格工程逻辑压出来的固定起点。

### 1. 汇总页先露出代表模板

在 `/workspace/resources` 汇总页中，每个大类都会展示一小组精选模板卡。

这样做的原因：

- 用户一进来就能确认“模板真的已经接进前端了”
- 汇总页不会显得只剩统计数字和跳转按钮
- 继续保留总览页的轻量结构，不会重新退回单页大杂烩

### 2. 模板卡优先

在子页里，每个模板都展示为一张独立卡片，卡片至少包含：

- 模板类型
- 模板所属分组
- 适合借用的场景摘要
- 推荐借用点
- 源码路径
- 如果是页面模板，则展示原始路由
- 如果命中推荐层级，则展示 `首选模板` 或 `团队推荐 Top N`

卡片操作入口现在拆成三类：

- `页面展示`：打开大尺寸可视化预览，先看这个模板整体更像什么页面或组件
- `源码拆解`：看结构说明、标签命中和依赖导入，理解为什么值得借
- `查看源码`：直接进入源码视图，不再和拆解视图混用

### 3. 可视化骨架预览

模板卡顶部不是截图，而是根据模板类型和标签生成的结构骨架预览。

这样做的原因：

- 上游仓库没有现成的页面截图资产
- 直接列文件名没有“模板感”
- 用结构骨架可以让开发者快速理解它是列表页、表单页、图表页还是工程模板

### 4. 详情弹窗查看源码块

点击模板后，会进入模板详情弹窗。

详情弹窗当前支持：

- 页面展示
- 概览
- Template 块
- Script 块
- Style 块
- Raw Code

同时还展示：

- 推荐借用点
- 推荐层级
- 团队推荐理由
- 依赖导入
- 相关模板
- 源码路径复制

## 当前指导思想

模板展厅的指导思想已经收敛为下面这套：

1. 不是“精选几张优秀页面”，而是“把上游前端按模板方式整库展示”
2. 不是只看页面，还要看组件和工程支撑
3. 不是只告诉开发者源码在哪，而要告诉开发者“这个模板适合借什么”
4. 默认入口必须可决策，所以要先做推荐模板和场景模板，再通过去重后的模板库补齐横向比较能力
5. 整个资源库还要额外钉死一组 `团队推荐 Top 10`，给开发者一个最稳定的默认起点
6. 迁移时优先借壳层、交互组织和拆分方式，不直接照搬上游业务语义
7. 模板库要自动收录，减少手工维护

## 后续扩展规则

后续继续往模板展厅里扩时，遵守下面规则：

1. 上游新模板优先通过自动收录进入展厅，不要继续手写台账
2. 新增模板分类时，优先保证“开发者能直接找得到”
3. 如果某类模板很重要，可以单独补专门的模板详情说明
4. 如果未来要做真实截图或 live preview，再在当前模板卡和详情页基础上增强，不要退回文件目录模式

## 当前价值

现在职责已经拆开：

- `资源总览` 负责统计、说明、导航和精选模板预览
- `资源总览` 还额外固定 `团队推荐 Top 10`
- `团队推荐 Top 10` 单独做成独立页面
- `页面模板 / 组件模板 / 工程模板` 三个子页内部再拆为 `推荐模板 / 场景模板 / 模板库`

这样做的价值是：

- 汇总页不会再被 260 个模板卡撑爆，但也不会空得像目录页
- 最常用的 10 个模板被钉死后，新开发者不会一进来就陷在同质化模板里
- 开发者能先拿到可直接开工的首选模板，而不是一上来就在重复卡片里打转
- 开发者能按任务类型进入场景模板，再决定要不要继续下潜
- 数据层仍然保留自动收录能力，但页面层只暴露去重后的模板集合，不需要再把整库原始档案直接摊给开发者
