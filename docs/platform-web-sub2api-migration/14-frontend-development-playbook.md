# 14. Frontend Development Playbook

## 目标

这份文档不是继续分析 `sub2api-base` 为什么好看，而是把我们已经收敛出来的前端开发范式正式固化下来。

从这一版开始，后续开发 `apps/platform-web` 的人默认遵守这份手册，不再需要每次回 `apps/platform-web-sub2api-base` 翻源码找感觉。

这份手册要解决四件事：

1. 新功能进入当前正式前端宿主 `platform-web` 时，应该按什么页面范式落地
2. 哪些组件、交互和视觉规则必须统一，不允许各搞一套
3. 什么时候借 `sub2api-base`，借什么，不借什么
4. 如何通过当前仓库内的 `Resources` 与 `examples` 找到标准模板和推荐母版

## 一、先记住三条总原则

### 1. 功能真相源来自当前正式宿主 `apps/platform-web`

页面是否存在、功能是否完整、链路是否对齐，一律以：

- `apps/platform-web`

为真相源。

当前 `platform-web` 的任务是：

- 在新的视觉与交互基座下完成正式迁移
- 把原有能力做得更美观、更工程化、更稳定

不是自由发明一个新的平台产品。

### 2. 视觉与交互基线固定为当前正式壳层 `apps/platform-web`

当前正式基线直接收口到：

- `apps/platform-web`

也就是说：

- 新功能必须长得像当前 `platform-web`
- 不能直接把 `sub2api-base` 的业务页整页搬进来
- 只能吸收它的壳层、交互组织和组件范式

### 3. `sub2api-base` 只作为参考源，不作为开发宿主

后续引用 `sub2api-base` 的唯一正确方式是：

- 借布局节奏
- 借组件拆分方式
- 借成熟交互母版
- 借工程封装模式

而不是：

- 直接拷贝它的业务语义
- 直接照搬它的路由和页面信息架构
- 把它当成我们的正式前端宿主

## 二、标准页面范式

### 1. 页面先选母版，不按灵感开发

新增页面前，先判断它属于哪一类：

1. 列表页
2. 详情页
3. 创建/编辑页
4. 工作区页
5. 资源/说明页

每类页面都应使用现有母版，而不是随手重新排版。

### 2. 列表页标准结构

默认结构固定为：

1. `PageHeader`
2. `StateBanner / GuidePanel` 可选提示区
3. 统计卡片 `MetricCard` 可选
4. 搜索/筛选区
5. 表格区 `DataTable`
6. 批量操作区 `BulkActionsBar` 可选
7. 分页区 `PaginationBar`

适用页面：

- `projects`
- `users`
- `assistants`
- `audit`
- `graphs`
- `testcase/cases`
- `testcase/documents`

列表页里不允许每个页面自己造：

- 搜索框样式
- 分页条样式
- 表格空态
- 排序交互
- 列设置入口

### 3. 详情页标准结构

默认结构固定为：

1. `PageHeader`
2. 摘要卡区
3. 分组内容区
4. 关联资源区
5. 最近活动 / 审计轨迹区

详情页要强调：

- 上下文清楚
- 概览先行
- 次级信息分区展示

不要把所有字段堆成一大页长表单。

### 4. 创建/编辑页标准结构

默认结构固定为：

1. `PageHeader`
2. 关键说明区
3. 表单主卡片
4. 可选附加配置卡片
5. 固定底部或页尾操作区

创建/编辑页里的字段编排必须遵守：

- 同类字段同组
- 次要说明放 hint，不塞进 label
- 危险操作与普通操作分区
- 长表单优先按 section 拆开

### 5. 工作区页标准结构

适用：

- `chat`
- `sql-agent`
- `runtime`
- `threads`
- `testcase/generate`

工作区页不按普通后台列表页处理，而是：

1. 顶部上下文说明
2. 主工作区
3. 侧栏/抽屉型辅助面板
4. 异步执行状态区
5. 错误、中断、引导和调试信息分层展示

这类页面最容易被做乱，必须遵守：

- 执行主路径永远优先
- 辅助信息放抽屉、弹窗或折叠区
- 不把所有调试信息常驻主界面

## 三、统一组件范式

### 1. 基础输入组件必须统一

表单输入必须优先复用：

- `BaseInput`
- `BaseSelect`
- `BaseButton`
- `BaseDialog`
- `ConfirmDialog`

不允许在业务页重新手写另一套：

- 输入框边框
- 下拉箭头样式
- 弹窗头尾结构
- 主次按钮风格

### 2. 表格体系必须统一

所有列表类页面优先走：

- `DataTable`
- `SearchInput`
- `FilterToolbar`
- `FilterSettingsMenu`
- `ActionMenu`
- `PaginationBar`

这套东西解决的是：

- 排序
- 列展示控制
- 批量选择
- 行级操作
- 空态
- 加载态
- 分页节奏

后续要继续向 `sub2api-base` 学的重点也在这：

- 长列表性能
- 移动端卡片降级
- 操作列自适应展开
- 跳页和更成熟的分页交互

### 3. 顶栏和下拉体系必须统一

顶部上下文条只允许复用和扩展：

- `TopContextBar`
- `WorkspaceProjectSwitcher`
- `LocaleSwitcher`
- `AnnouncementCenter`
- `UserMenu`

顶部控件的要求是：

- 尺寸统一
- 圆角统一
- 背景一致
- 下拉层级一致
- hover 和 open 状态一致
- 项目切换、语言切换、公告中心、用户菜单使用同一套触发器和 dropdown 语法
- 不允许为了某一个业务入口临时在顶栏搓一套新按钮和新弹层

不允许不同页面在顶栏临时拼接一组风格不一样的按钮。

### 4. 状态反馈必须统一

状态反馈优先使用：

- `StateBanner`
- `EmptyState`
- `StatusPill`
- `ToastStack`
- `NavigationProgress`

任何空页面、无数据页面、接口失败页面，不允许只丢一行裸文案。

## 四、视觉与主题规则

### 1. 不允许页面自由决定视觉语言

页面不应该自己决定：

- 新的主色
- 新的按钮体系
- 新的圆角体系
- 新的卡片背景语言

这些都必须沿用：

- `tailwind.config`
- `src/styles/index.css`
- 已有基础组件

### 2. 深浅色模式必须同时成立

后续新增组件和页面时，必须同时检查：

1. 默认浅色是否成立
2. 深色模式是否可读
3. 下拉、弹窗、表格 hover 是否一致
4. 首屏是否会闪主题

深色模式不能等页面全做完再补。

### 3. 响应式不是最后再补

新增页面和公共组件时，必须同时检查：

1. 手机宽度下顶栏是否会换行而不是把页面撑爆
2. 筛选栏是否允许 `wrap`，固定宽度控件是否会退化成 `w-full`
3. 左右双栏在窄屏下是否能自然折叠，而不是继续死撑固定列宽
4. 弹层、drawer、分页条、聊天输入区是否会遮挡或超出视口

公共壳层和公共组件要优先解决这些问题，不能把响应式压力继续甩给业务页。

### 4. 色彩使用克制

当前风格已经收敛，不再追求“AI 风格”的过度发光和大面积饱和色。

色彩策略固定为：

- 主色只负责重点
- 状态色只负责状态
- 大面积背景保持冷静和克制
- 强视觉只用在摘要、引导和极少数关键 CTA

## 五、国际化与文案规则

### 1. 共享壳层文案必须进入 i18n

以下内容必须进入 i18n：

- 导航
- 顶栏
- 通用按钮
- 分页
- Toast
- 通用空态

业务深层文案可以分阶段推进，但共享壳层不允许继续写死。

### 2. 页面内文案先保证结构清晰，再谈润色

文案表达要求：

- 标题清楚
- 说明简洁
- hint 可执行
- 危险操作明确

不要把说明写成大段废话，也不要用“技术人自我感动式”文案。

## 六、什么时候看 Resources，什么时候看文档

### 1. 要看“怎么做”，先看文档

如果开发者要判断：

- 一个新页面该选哪种结构
- 该复用哪些组件
- 哪些规则不能破

先看：

- `14-frontend-development-playbook.md`

### 2. 要看“长什么样”，先看 Resources

如果开发者要判断：

- 列表页参考谁
- 顶栏 dropdown 参考谁
- 表格、分页、弹窗、状态块怎么借

先看：

- `/workspace/resources`
- `/workspace/resources/playbook`
- `/workspace/resources/top-picks`
- `/workspace/resources/pages`
- `/workspace/resources/components`
- `/workspace/resources/engineering`

## 七、Resources 的正确用法

Resources 不是源码档案馆，而是开发者工作台。

正确使用顺序：

1. 先看 `开发范式`
2. 再看 `团队推荐 Top 10`
3. 需要页面骨架时看 `页面模板`
4. 需要局部交互时看 `组件模板`
5. 需要接线方式时看 `工程模板`

不建议直接从完整模板库开始翻。

## 八、后续开发执行清单

每开发一个新功能，默认按下面顺序执行：

1. 先确认 `apps/platform-web` 是否已有该页面或能力
2. 判断页面类型，选母版
3. 先从 `Resources` 找参考模板
4. 优先复用现有基础组件和平台组件
5. 新增组件时先想是否属于通用层
6. 同时补齐浅色 / 深色
7. 共享壳层文案同步进入 i18n
8. 自查空态、加载态、错误态
9. 自查项目上下文、权限和工作区联动
10. 最后再做视觉微调

## 九、当前明确要继续向 `sub2api-base` 学的部分

当前最值得继续收口的能力是：

1. 更成熟的 `Select` 能力
2. 更成熟的 `DataTable` 交互与长列表承载
3. 更完整的 `Pagination`
4. 更统一的顶栏 dropdown 体系
5. 更标准化的“创建/编辑/属性配置”弹窗与表单模式

这些能力收口后，`platform-web` 后续新增页面的工程质量会明显更稳。

## 十、结论

从这版开始，团队默认遵守这条开发口径：

- 功能对齐看 `platform-web`
- 风格与交互看 `platform-web`
- 视觉资产参考看 `sub2api-base`
- 规范和模板入口统一看当前仓库里的文档与 `Resources`

这件事的最终目标不是“少翻几个目录”，而是让后续开发者在当前仓库里就能完成：

- 理解规范
- 找到模板
- 复用组件
- 按统一母版交付页面
