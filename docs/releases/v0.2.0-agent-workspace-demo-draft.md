# v0.2.0 - 容器化交付可用版

> 正式发布说明草案，可直接用于 GitHub Release。

## Summary

`v0.2.0` 是 `v0.1.2` 之后的一轮交付面扩展版本。
这一版把仓库从“本地脚本可演示”推进到“容器方式可直接 bring-up”，补齐 `runtime-service` 单应用容器方案、整仓 compose（无 Nginx / 带 Nginx）方案、共享 Postgres 初始化脚本、operator 文档入口，以及容器环境下真实暴露出来的一组联调缺口修复。

核心结果是：

- 用户可以按文档直接启动单应用 `runtime-service`
- 用户可以按文档直接启动整仓 stack，并完成前端登录、runtime 目录刷新与基础健康验证
- 异步 worker、远端持久化、多模态解析默认值这些“容器化后才暴露”的配置漂移问题已收敛

## Highlights

- 新增 `runtime-service` 单应用 Docker 交付面，支持 `docker compose build` / `up -d`
- 新增整仓 compose 无 Nginx / 带 Nginx 两种部署方式，并补齐零到一 operator 文档
- 修复 no-nginx 前端 API 指向、`localhost` / `127.0.0.1` CORS、worker upstream 漂移等容器化主链路问题
- 将容器化共享多模态附件解析默认模型收敛为 `gpt_5.4-ccr`
- 打通 `runtime-service -> interaction-data-service` 的远端持久化配置，避免 testcase v2 错报“远端未配置”

## Added

- 新增 `apps/runtime-service/deploy/` 交付资产：
  - `Dockerfile`
  - `docker-compose.runtime-service.yml`
  - `.env.runtime-service.example`
  - `README.md`
- 新增整仓容器化交付资产：
  - `deploy/docker-compose.stack.yml`
  - `deploy/docker-compose.stack.nginx.yml`
  - `deploy/nginx/default.conf`
  - `deploy/postgres/init/01-init-shared-databases.sh`
  - `deploy/.env.stack.example`
- 新增容器化 operator 文档：
  - `deploy/README.md`
  - `docs/zero-to-one-container-deploy.md`
  - `docs/runbooks/container-update-runbook.md`
  - `docs/container-address-guide.md`
- 新增 `platform-web-vue` 针对 no-nginx API base 行为的回归测试

## Changed

- 将根目录 README 补齐 Docker / Docker Compose 的 3 条实际 bring-up 命令
- 将部署说明、环境矩阵与 docs 总入口收敛到新的容器化文档路径
- 将容器化共享多模态附件解析默认值收敛为 `MULTIMODAL_PARSER_MODEL_ID=gpt_5.4-ccr`
- 为 `interaction-data-service`、`platform-web-vue`、`runtime-service` 增补更轻量的构建上下文与生产容器配置
- 将 `interaction-data-service` 容器健康检查改成 Python 内建探活，避免镜像仅为 `curl` / 系统工具链拉取大包

## Fixed

- 修复 no-nginx 前端把显式 `http://localhost:2142` 误改写成同源 `/api` 的问题
- 修复 `localhost:3000` 与 `127.0.0.1:3000` 入口不一致导致的 CORS 失败
- 修复 `platform-api-v2-worker` 未同步主服务 upstream env，导致 runtime catalog 异步刷新失败
- 修复 `runtime-service` 容器未配置 `INTERACTION_DATA_SERVICE_URL`，导致 testcase v2 持久化误报“远端未配置”
- 修复 `platform-api-v2` 的 operation duration 汇总在 PostgreSQL 下的 SQL 方言不兼容问题

## Verification

- `pnpm --dir apps/platform-web-vue exec vitest run src/config/env.spec.ts`
- `docker compose -f deploy/docker-compose.stack.yml --env-file deploy/.env.stack config`
- `docker compose -f deploy/docker-compose.stack.nginx.yml --env-file deploy/.env.stack config`
- 浏览器 / Playwright 验证：
  - `http://localhost:3000` 登录成功
  - `http://127.0.0.1:3000` 登录成功
  - `runtime/models` 目录刷新成功
  - `runtime/tools` 目录刷新成功
- 容器内链路验证：
  - `platform-api-v2-worker -> runtime-service`
  - `platform-api-v2-worker -> interaction-data-service`
  - `platform-api-v2-worker -> LightRAG /host.docker.internal`
  - `runtime-service -> interaction-data-service`
  - `runtime-service` 多模态解析默认模型已切到 `gpt_5.4-ccr`

## Known Issues

- `test_case_service_v2` 的前端真实对话到远端持久化 happy path 仍建议再做一轮浏览器级端到端验收
- 公网域名、局域网 IP、非 localhost ingress 仍需要按文档同步调整前端 URL 与 CORS 白名单
- `runtime-web` 仍是调试入口，不是正式产品主前端

## Recommended Demo Flow

建议对外演示按下面顺序展开：

1. 根目录 README 的 Docker 方案入口
2. `deploy/README.md`
3. `docs/zero-to-one-container-deploy.md`
4. 启动无 Nginx 整仓 stack
5. 前端登录
6. Runtime Models / Tools 目录刷新
7. Testcase / Knowledge 基础页面加载
