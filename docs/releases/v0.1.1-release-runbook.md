# v0.1.1 发版执行清单

## 1. 目标

这份清单用于准备下一阶段版本 `v0.1.1`，避免到时又靠人脑硬记。

当前默认口径：

- 版本号：`v0.1.1`
- 发版对象：整个仓库
- 发版基线：`main`
- 版本口径：`Agent 工作台可演示版`

## 2. 参考文档

发版前先看下面这些文档，别瞎打 tag：

- [release-policy.md](./release-policy.md)
- [v0.1.1-agent-workspace-demo-draft.md](./v0.1.1-agent-workspace-demo-draft.md)
- [CHANGELOG.md](../CHANGELOG.md)
- [16-smoke-test-and-acceptance-checklist.md](../platform-web-sub2api-migration/16-smoke-test-and-acceptance-checklist.md)

## 3. 发版前准备

至少确认下面这些事：

- 计划进入 `v0.1.1` 的功能已经合并到 `main`
- `docs/CHANGELOG.md` 的 `Unreleased` 已经补齐
- `v0.1.1-agent-workspace-demo-draft.md` 已补到可直接发出的程度
- 演示链路和账号仍然可用
- 已知问题已经归档

## 4. 变更归档

建议以 `v0.1.0..main` 为范围整理本轮变更：

```bash
git log --oneline v0.1.0..main
```

如果需要进一步提炼正文，可结合：

```bash
git log --format=medium v0.1.0..main
```

整理时重点只收：

- `feat`
- `fix`
- `perf`

然后把结果回填到：

- `docs/CHANGELOG.md`
- `docs/releases/v0.1.1-agent-workspace-demo-draft.md`

## 5. main 上的最终校验

正式发版前至少执行：

```bash
pnpm --dir "apps/platform-web-vue" lint
pnpm --dir "apps/platform-web-vue" build
```

推荐再按烟测清单手动走一遍关键路由。

## 6. 打 tag

确认 `main` 已经是最终发布状态后：

```bash
git checkout main
git pull origin main
git tag -a v0.1.1 -m "release: v0.1.1"
```

检查 tag：

```bash
git tag --list | rg "^v0\.1\.1$"
git show v0.1.1 --stat --no-patch
```

## 7. 推送 main 与 tag

```bash
git push origin main
git push origin v0.1.1
```

确认远端 tag：

```bash
git ls-remote --tags origin | rg "refs/tags/v0\.1\.1$"
```

## 8. 创建 GitHub Release

如果使用 `gh`：

```bash
gh release create v0.1.1 \
  --draft \
  --title "v0.1.1 - Agent 工作台可演示版" \
  --notes-file "docs/releases/v0.1.1-agent-workspace-demo-draft.md"
```

如果走网页创建，也优先直接复用同一个草稿文档。

## 9. 发版后回写

完成发布后，至少确认：

- `docs/CHANGELOG.md` 已形成正式版本条目
- GitHub Release 已从 draft 发布
- 团队同步信息里附上正式 release 链接
- 下一轮未发布内容重新归到 `Unreleased`
