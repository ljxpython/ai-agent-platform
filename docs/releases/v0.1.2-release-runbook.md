# v0.1.2 发版执行清单

## 1. 目标

这份清单用于准备 `v0.1.2` 发版，避免到时又靠人脑硬记。

当前默认口径：

- 版本号：`v0.1.2`
- 发版对象：整个仓库
- 发版基线：`main`
- 版本口径：`Agent 工作台可演示版`

## 2. 参考文档

发版前先看下面这些文档，别瞎打 tag：

- [release-policy.md](./release-policy.md)
- [v0.1.2-agent-workspace-demo-draft.md](./v0.1.2-agent-workspace-demo-draft.md)
- [CHANGELOG.md](../CHANGELOG.md)
- [16-smoke-test-and-acceptance-checklist.md](../platform-web-sub2api-migration/16-smoke-test-and-acceptance-checklist.md)

## 3. 发版前准备

至少确认下面这些事：

- 计划进入 `v0.1.2` 的功能已经合并到 `main`
- `docs/CHANGELOG.md` 的 `Unreleased` 已经提炼完成
- `v0.1.2-agent-workspace-demo-draft.md` 已补到可直接发出的程度
- 演示链路和账号仍然可用
- 已知问题已经归档
- 版本号对齐工作已经完成（至少确认前端展示版本、控制面 API 版本和仓库 release tag 不冲突）

## 4. 变更归档

建议以 `v0.1.1..main` 为范围整理本轮变更：

```bash
git log --oneline v0.1.1..main
```

如果需要进一步提炼正文，可结合：

```bash
git log --format=medium v0.1.1..main
```

整理时优先提炼：

- 用户可感知的新能力
- 演示链路上的体验变化
- 必须对外说明的修复项

若提交仍保留 Conventional Commits 类型，可继续优先收：

- `feat`
- `fix`
- `perf`

然后把结果回填到：

- `docs/CHANGELOG.md`
- `docs/releases/v0.1.2-agent-workspace-demo-draft.md`

## 5. main 上的最终校验

正式发版前至少执行当前 CI 对齐校验：

```bash
cd apps/platform-api-v2
uv sync
uv run python -m compileall app
uv run python -m unittest discover -s tests -p 'test*.py' -v

cd ../platform-web-vue
pnpm install --frozen-lockfile
pnpm lint
pnpm typecheck
pnpm build
```

推荐补充执行本轮重点变更相关校验：

```bash
pytest -q apps/interaction-data-service/tests/test_test_case_service_api.py

cd apps/runtime-service
uv run pytest runtime_service/tests/test_multimodal_defaults.py \
  runtime_service/tests/test_multimodal_middleware.py \
  runtime_service/tests/test_research_agent_graph.py \
  runtime_service/tests/test_test_case_service_v2_graph.py -q
```

推荐再执行一轮本地 bring-up 与健康检查：

```bash
scripts/dev-up.sh
scripts/check-health.sh
```

并按烟测清单手动走一遍关键路由，尤其复查：

- `/workspace/projects`
- `/workspace/chat`
- `/workspace/threads`
- `/workspace/testcase/generate`
- `/workspace/testcase/cases`
- `/workspace/testcase/documents`
- 项目级 knowledge 相关页面

## 6. 打 tag

确认 `main` 已经是最终发布状态后：

```bash
git checkout main
git pull origin main
git tag -a v0.1.2 -m "release: v0.1.2"
```

检查 tag：

```bash
git tag --list | rg "^v0\.1\.2$"
git show v0.1.2 --stat --no-patch
```

## 7. 推送 main 与 tag

```bash
git push origin main
git push origin v0.1.2
```

确认远端 tag：

```bash
git ls-remote --tags origin | rg "refs/tags/v0\.1\.2$"
```

## 8. 创建 GitHub Release

如果使用 `gh`：

```bash
gh release create v0.1.2 \
  --draft \
  --title "v0.1.2 - Agent 工作台可演示版" \
  --notes-file "docs/releases/v0.1.2-agent-workspace-demo-draft.md"
```

如果走网页创建，也优先直接复用同一个草稿文档。

## 9. 发版后回写

完成发布后，至少确认：

- `docs/CHANGELOG.md` 已形成正式版本条目
- GitHub Release 已从 draft 发布
- 团队同步信息里附上正式 release 链接
- 下一轮未发布内容重新归到 `Unreleased`
- 发版后默认版本目标已切到下一轮里程碑
