# v0.2.0 发版执行清单

## 1. 目标

这份清单用于准备 `v0.2.0` 发版，避免到时又靠人脑硬记。

当前默认口径：

- 版本号：`v0.2.0`
- 发版对象：整个仓库
- 发版基线：`main`
- 版本口径：`容器化交付可用版`

## 2. 参考文档

发版前先看下面这些文档，别瞎打 tag：

- [release-policy.md](./release-policy.md)
- [v0.2.0-agent-workspace-demo-draft.md](./v0.2.0-agent-workspace-demo-draft.md)
- [CHANGELOG.md](../CHANGELOG.md)
- [deploy/README.md](../../deploy/README.md)
- [zero-to-one-container-deploy.md](../zero-to-one-container-deploy.md)
- [container-update-runbook.md](../runbooks/container-update-runbook.md)

## 3. 发版前准备

至少确认下面这些事：

- 计划进入 `v0.2.0` 的功能已经合并到 `main`
- `docs/CHANGELOG.md` 的 `Unreleased` 已经提炼完成
- `v0.2.0-agent-workspace-demo-draft.md` 已补到可直接发出的程度
- 单应用 `runtime-service` 容器方案可 build / up / health
- 整仓 compose 无 Nginx 可登录并走通基础 smoke
- 整仓 compose 带 Nginx 可启动并通过入口健康检查
- 已知问题已经归档
- 版本号对齐工作已经完成

## 4. 变更归档

建议以 `v0.1.2..main` 为范围整理本轮变更：

```bash
git log --oneline v0.1.2..main
```

如果需要进一步提炼正文，可结合：

```bash
git log --format=medium v0.1.2..main
```

整理时优先提炼：

- 用户可直接使用的新交付方式
- 容器 bring-up / 验证链路上的修复
- 必须对外说明的配置口径

然后把结果回填到：

- `docs/CHANGELOG.md`
- `docs/releases/v0.2.0-agent-workspace-demo-draft.md`

## 5. main 上的最终校验

正式发版前至少执行当前对齐校验：

```bash
pnpm --dir apps/platform-web-vue exec vitest run src/config/env.spec.ts
docker compose -f deploy/docker-compose.stack.yml --env-file deploy/.env.stack config
docker compose -f deploy/docker-compose.stack.nginx.yml --env-file deploy/.env.stack config
```

推荐补充执行：

```bash
docker compose -f apps/runtime-service/deploy/docker-compose.runtime-service.yml --env-file apps/runtime-service/deploy/.env.runtime-service config
docker compose -f apps/runtime-service/deploy/docker-compose.runtime-service.yml --env-file apps/runtime-service/deploy/.env.runtime-service up -d
curl http://127.0.0.1:8123/info
curl http://127.0.0.1:8123/internal/capabilities/models
curl http://127.0.0.1:8123/internal/capabilities/tools
```

以及整仓 smoke：

```bash
docker compose -f deploy/docker-compose.stack.yml --env-file deploy/.env.stack up -d
curl http://127.0.0.1:8081/_service/health
curl http://127.0.0.1:2142/_system/probes/ready
curl http://127.0.0.1:8123/info
curl -I http://localhost:3000
```

推荐再手动验证：

- `http://localhost:3000`
- `http://127.0.0.1:3000`
- 登录
- `/workspace/runtime/models`
- `/workspace/runtime/tools`
- `Testcase / Knowledge` 页面壳层加载

## 6. 打 tag

确认 `main` 已经是最终发布状态后：

```bash
git checkout main
git pull origin main
git tag -a v0.2.0 -m "release: v0.2.0"
```

检查 tag：

```bash
git tag --list | rg "^v0\.2\.0$"
git show v0.2.0 --stat --no-patch
```

## 7. 推送 main 与 tag

```bash
git push origin main
git push origin v0.2.0
```

确认远端 tag：

```bash
git ls-remote --tags origin | rg "refs/tags/v0\.2\.0$"
```

## 8. 创建 GitHub Release

如果使用 `gh`：

```bash
gh release create v0.2.0 \
  --draft \
  --title "v0.2.0 - 容器化交付可用版" \
  --notes-file "docs/releases/v0.2.0-agent-workspace-demo-draft.md"
```

如果走网页创建，也优先直接复用同一个草稿文档。

## 9. 发版后回写

完成发布后，至少确认：

- `docs/CHANGELOG.md` 已形成正式版本条目
- GitHub Release 已从 draft 发布
- README / deploy 文档中的版本说明如有必要已更新
- 下一轮未发布内容重新归到 `Unreleased`
- 发版后默认版本目标已切到下一轮里程碑
