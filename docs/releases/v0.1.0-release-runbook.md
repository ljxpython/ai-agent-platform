# v0.1.0 发版执行清单

## 1. 目标

这份清单用于把当前仓库的第一次正式 release 走顺，不靠临场记忆。

适用目标：

- 版本号：`v0.1.0`
- 发版对象：整个仓库
- 发版基线：`main`
- 版本口径：`Agent 工作台可演示版`

## 2. 发版前确认

在开始任何 Git 操作前，先确认下面这些事：

- 需要发版的功能分支都已经准备好
- 当前演示主线已经能完整复述
- 关键文档已经齐全
- 本地环境可完成最小校验

当前最重要的参考文档：

- [release-policy.md](./release-policy.md)
- [v0.1.0-agent-workspace-demo-draft.md](./v0.1.0-agent-workspace-demo-draft.md)
- [16-smoke-test-and-acceptance-checklist.md](../platform-web-sub2api-migration/16-smoke-test-and-acceptance-checklist.md)

## 3. 合并到 main

如果当前功能在功能分支，比如：

- `feat/platform-web-frontend-optimization`

正确流程是先合并到 `main`，再从 `main` 发版。

推荐操作：

```bash
git checkout main
git pull origin main
git merge --no-ff feat/platform-web-frontend-optimization
```

如果是通过 GitHub PR 合并，也满足要求。

关键点：

- 不要从 feature 分支直接打 tag
- 不要在未合并到 `main` 的状态下创建 GitHub Release

## 4. main 上的最终校验

切到 `main` 后，执行：

```bash
pnpm --dir "apps/platform-web-vue" lint
pnpm --dir "apps/platform-web-vue" build
```

再按烟测清单手动走关键路由：

- `/workspace/overview`
- `/workspace/assistants`
- `/workspace/chat`
- `/workspace/sql-agent`
- `/workspace/threads`
- `/workspace/testcase/generate`
- `/workspace/testcase/cases`
- `/workspace/testcase/documents`
- `/workspace/announcements`
- `/workspace/resources/playbook`

账号确认：

- `admin / admin123456`
- `test / test`

如果任何一条主链路没过，不要继续打 tag。

## 5. 更新 CHANGELOG

正式 release 前，建议把 `docs/CHANGELOG.md` 从：

```md
## [Unreleased]
```

整理为包含 `v0.1.0` 的版本条目。

推荐直接参考：

- [v0.1.0-agent-workspace-demo-draft.md](./v0.1.0-agent-workspace-demo-draft.md)

建议结构：

- `Added`
- `Changed`
- `Fixed`

## 6. 打 tag

在确认 `main` 已经是最终发布状态后：

```bash
git checkout main
git pull origin main
git tag -a v0.1.0 -m "release: v0.1.0"
```

检查 tag：

```bash
git tag --list | grep "v0.1.0"
git show v0.1.0 --stat
```

## 7. 推送 main 与 tag

```bash
git push origin main
git push origin v0.1.0
```

确认远端 tag 已存在：

```bash
git ls-remote --tags origin | grep "v0.1.0"
```

## 8. 创建 GitHub Release

进入仓库 GitHub 页面：

1. 打开 `Releases`
2. 点击 `Draft a new release`
3. 选择 tag：`v0.1.0`
4. 标题填写：

```text
v0.1.0 - Agent 工作台可演示版
```

5. 正文内容优先使用：

- [v0.1.0-agent-workspace-demo-draft.md](./v0.1.0-agent-workspace-demo-draft.md)

如果需要补充，可以增加：

- 已知问题
- 部署说明
- 演示账号

## 9. 发版后回写

完成 GitHub Release 后，建议同步做这几件事：

1. 确认 `docs/CHANGELOG.md` 已提交到仓库
2. 在团队群或项目同步里发 release 链接
3. 在后续迭代中把新开发默认归到下一版本，例如：
   - `v0.1.1`
   - `v0.2.0`

## 10. 一次完整示例

如果当前功能分支已经准备好，一套完整命令大概是：

```bash
git checkout main
git pull origin main
git merge --no-ff feat/platform-web-frontend-optimization

pnpm --dir "apps/platform-web-vue" lint
pnpm --dir "apps/platform-web-vue" build

git tag -a v0.1.0 -m "release: v0.1.0"
git push origin main
git push origin v0.1.0
```

然后去 GitHub 页面创建：

- `v0.1.0 - Agent 工作台可演示版`

## 11. 本次 release 的出厂标准

只要满足下面这些点，就可以认为这次 release 是合格的：

- `main` 上代码和文档一致
- tag 正确打在 `main`
- GitHub Release 已创建
- CHANGELOG 已更新
- 演示账号、烟测、已知问题都能找到正式文档
