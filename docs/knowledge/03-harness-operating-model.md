# Harness Operating Model（AITestLab 的运行方式与落地操作模型）

文档类型：`Knowledge Operating Model`

本文解释团队和 AI 代理应如何在当前 harness 中工作。它描述的是操作模型，不直接替代某个服务的 README、contract 或 API 文档。

## 这篇文档解决什么问题

前两篇文档已经把两件事讲清楚了：

- `01-harness-engineering-foundation.md`：Harness Engineering 是什么
- `02-aitestlab-harness-blueprint.md`：它在当前仓库里落在哪些层、哪些目录、哪些服务上

但工程要真正跑起来，还缺最关键的一块：

> 团队和 AI 代理在日常开发中，到底应该按什么动作顺序工作？

所以这篇文档专门回答“怎么做”。

它不是抽象理念文档，而是一份操作模型，重点解决下面这些问题：

- 一个需求从进入仓库到完成交付，标准动作是什么
- 文档、代码、测试、runbook 应该在什么时候补
- 什么叫完成，什么叫没完成
- 什么情况下应该停下来先补契约和验收，而不是继续写代码
- 团队如何把 Harness 从“知道”变成“每天照着做”

---

## 1. Operating Model 的核心定义

对当前仓库，推荐把 Operating Model 定义成一句话：

> 任何改动，都必须先被定位到正确层级，再按“契约 -> 设计 -> 实现 -> 验收 -> 排障 -> 交付”的顺序推进。

这个顺序不能反。

很多仓库一乱，根本原因就一句话：

- 契约还没定，就开始写代码
- 验收还没想清，就开始堆功能
- 出问题没有缩圈路径，只能靠猜
- 交付时才发现文档、测试、runbook 都没跟上

所以，Harness Operating Model 的第一原则不是“快写”，而是：

> 先把动作顺序固定住，再追求速度。

---

## 2. AITestLab 的标准工作流

建议把所有工作都纳入下面这条主流程：

```text
需求进入
-> 层级判定
-> 契约澄清
-> 设计落点
-> 实现
-> 本地验证
-> 链路验证
-> 文档与 runbook 补齐
-> 交付 / 合入
```

下面逐步展开。

---

## 3. 第一步：层级判定（先判断这事该谁干）

任何需求开始前，先做层级判定。

### 3.1 必须先回答的问题

1. 这是平台治理问题，还是 Runtime 执行问题？
2. 这是调试能力问题，还是正式产品链路问题？
3. 这是结果域持久化问题，还是 Agent 行为问题？
4. 这是仓库级规则问题，还是服务内部实现问题？

### 3.2 当前仓库的典型归类方式

- `apps/platform-web-vue`：页面宿主、工作区、平台前端交互
- `apps/platform-api-v2`：平台治理、网关、鉴权、项目边界
- `apps/runtime-service`：graph、middleware、runtime contract、model、tools、MCP
- `apps/runtime-web`：Runtime 调试 UI
- `apps/interaction-data-service`：结果持久化与查询
- 根目录 `docs/`：仓库级知识、流程、部署、发布、范式

### 3.3 层级判定的硬规则

如果一个需求还说不清属于哪一层，**不应该直接开写代码**。

先补清楚：

- 功能归属
- 输入归属
- 验收归属
- 排障归属

不然写着写着就跨层污染，最后所有服务一起烂。

---

## 4. 第二步：契约澄清（先定输入与边界）

层级判定完，第二步不是写实现，而是先定契约。

### 4.1 契约澄清最少要写清的东西

1. **公开输入是什么**
2. **默认值从哪里来**
3. **哪些字段禁止传**
4. **非法输入如何失败**
5. **哪些是业务输入，哪些是执行控制，哪些是平台私有字段**

### 4.2 对当前项目的典型契约判断

以 `runtime-service` 为例，最典型的就是：

- `RuntimeContext`：公共业务输入
- `config`：执行控制
- `config.configurable`：线程 / 平台 / 私有字段
- `env`：部署默认值与 secrets

### 4.3 契约没清楚时的停止条件

出现下面任一情况，应该暂停实现，先补契约：

- 一个字段不知道该进 `context` 还是 `configurable`
- 一个需求需要从多个来源兜底乱读
- 一个 graph 准备自己发明一套参数读取方式
- 一个“兼容入口”开始向正式范式渗透
- 工具、模型、prompt 的动态控制边界说不清

这一步越偷懒，后面返工越惨。

---

## 5. 第三步：设计落点（决定该放哪、怎么放）

契约确定后，再做设计落点。

### 5.1 设计时必须写清四件事

1. **落在哪个目录 / 服务 / 模块**
2. **依赖哪些现行标准**
3. **明确不做什么**
4. **怎么验收**

### 5.2 推荐使用的设计描述模板

每个较大变更，至少写清：

```text
目标：
边界：
输入契约：
目录落点：
不做清单：
验收标准：
排障入口：
```

### 5.3 当前仓库的设计落点原则

- 仓库级规则：进根目录 `docs/`
- Runtime 范式：进 `apps/runtime-service/runtime_service/docs/standards/`
- Runtime 学习资料：进 `apps/runtime-service/runtime_service/docs/knowledge/`
- Runtime 排障：进 `apps/runtime-service/runtime_service/docs/runbooks/`
- 仓库级思想方法：进根目录 `docs/knowledge/`
- 调试前端通用逻辑：进 `apps/runtime-web`
- 平台治理逻辑：进 `apps/platform-api-v2`

---

## 6. 第四步：实现（按最小可持续方式落代码）

到了实现阶段，核心原则不是“能跑就行”，而是：

> 用最小复杂度，把现行契约和标准表达出来。

### 6.1 实现阶段的推荐原则

- 优先复用现有标准，不重新发明概念
- 优先静态结构，不默认上重动态
- 优先补共享层，不在各 graph / 各页面里重复手搓
- 优先直接报错，不做静默吞错
- 优先薄实现，不提前做未来能力预留

### 6.2 当前仓库里的典型实现判断

#### 对 `runtime-service`

优先顺序应是：

1. 先看 `docs/standards/`
2. 再看 `docs/knowledge/`
3. 决定是否应该落到：
   - graph
   - middleware
   - tools registry
   - runtime contract
   - service 模块
4. 最后再写代码

#### 对 `runtime-web`

优先保证：

- 通用调试能力
- 通用 context 传递
- 不 graph-specific
- 不替正式产品链路背锅

#### 对平台层

优先保证：

- 治理边界清晰
- Runtime 网关职责单一
- 页面宿主与工作台结构稳定

---

## 7. 第五步：本地验证（先证明没有把自己写炸）

任何实现完成后，先做本地验证。

### 7.1 本地验证至少包括

1. 语法正确
2. 构建通过
3. 目标服务测试通过
4. 必要的 harness tests 通过

### 7.2 验证顺序建议

```text
静态检查
-> 单测 / harness tests
-> 最小服务 smoke
-> 再做跨服务联调
```

### 7.3 为什么不能一上来就联调整体链路

因为整体链路一旦挂了，定位范围太大。

应该先证明：

- 我改的服务本身没炸
- 结构规则没破
- 输入契约仍然成立

再去看跨服务协作。

---

## 8. 第六步：链路验证（证明它不只是局部自嗨）

本地验证之后，再做链路验证。

### 8.1 链路验证分两类

#### A. 调试链路验证

```text
runtime-web -> runtime-service
```

适合验证：

- Runtime 本体行为
- context 透传
- graph / middleware / tools 行为
- 调试交互是否正常

#### B. 正式平台链路验证

```text
platform-web-vue -> platform-api-v2 -> runtime-service -> interaction-data-service
```

适合验证：

- 网关注入
- 项目边界
- 平台治理逻辑
- 页面工作流
- 结果域承接

### 8.2 推荐验证顺序

先调试链路，再正式链路。

原因很简单：

- 调试链路更短
- 问题范围更小
- 更容易把 Runtime 问题和平台问题分开

---

## 9. 第七步：文档与 Runbook 补齐（别等以后）

很多人把文档更新拖到最后，最后就是不更新。

当前仓库要反过来：

> 只要改动改变了范式、契约、排障路径、验收方式，就必须同步补文档。

### 9.1 什么时候必须补文档

- 新增或收紧运行时契约
- 修改 graph / middleware / tool registry 范式
- 改变调试方式
- 改变联调方式
- 改变错误定位方式
- 改变发布前检查要求

### 9.2 什么时候必须补 runbook

- 故障缩圈路径变了
- 某类错误现在有稳定排查方法了
- 新增了重要 smoke / health check 手段
- 新增了典型失败场景与恢复动作

### 9.3 文档补齐的硬规则

如果实现已经变了，但文档和 runbook 没跟上，这个工作就**不算完成**。

---

## 10. 第八步：交付 / 合入（以可验收状态结束）

最后一步不是“代码写完”，而是“交付闭环成立”。

### 10.1 一次改动的 Done Definition（推荐版）

一个改动至少满足下面这些条件，才算 Done：

1. 层级归属清楚
2. 契约边界清楚
3. 代码落点正确
4. 本地验证通过
5. 最小链路验证通过
6. 文档已更新
7. runbook 已更新（若受影响）
8. 遗留项和不做项已写清

### 10.2 不能算 Done 的常见情况

- 代码能跑，但契约还是乱的
- 代码能跑，但只有作者自己知道怎么测
- 代码能跑，但文档还是旧的
- 代码能跑，但出了问题没人知道去哪查
- 代码能跑，但绕过了现行范式

这种都不叫完成，这叫埋雷。

## 11. AITestLab 的角色分工建议

Harness Operating Model 不是只给人看的，也应该给 AI 代理看。

### 11.1 人类开发者的职责

重点负责：

- 判断需求层级
- 确认边界和取舍
- 决定不做什么
- 决定正式验收口径
- 对架构走向负责

### 11.2 AI 代理的职责

重点负责：

- 按既有标准落实现
- 按既有文档补知识和模板
- 执行结构化检查与验证
- 产出 smoke 结果、差异说明、遗留项总结
- 协助把“隐性经验”变成显性文档

### 11.3 两者的协作原则

推荐采用这种分工：

- **人决定方向和边界**
- **AI 执行实现和结构化收尾**

一旦边界都没定，就把实现全扔给 AI，最后大概率就是一坨屎山。

## 12. 模板：一个标准变更应如何推进

下面给一版最小操作模板，后续任何中等以上改动都可以照着走。

### 12.1 变更启动模板

```text
【变更主题】

1. 目标
2. 所属层级
3. 受影响服务 / 目录
4. 输入契约
5. 不做清单
6. 验收标准
7. 排障入口
```

### 12.2 实现完成模板

```text
【实现结果】

1. 实际改动点
2. 契约是否变化
3. 文档是否更新
4. runbook 是否更新
5. 本地验证结果
6. 链路验证结果
7. 已知遗留项
```

### 12.3 故障复盘模板

```text
【故障复盘】

1. 现象
2. 影响范围
3. 根因层级（平台 / runtime / 调试 / 数据 / 契约）
4. 定位路径
5. 修复动作
6. 是否需要补标准 / runbook / harness tests
```

## 13. 哪些信号说明 Harness 在变好

你别只看功能数量，判断 Harness 是否变强，应该看这些信号：

1. 新需求更容易判断落点
2. 参数归位争议越来越少
3. 调试链路和正式链路更容易区分
4. 排障时能快速缩圈，而不是到处猜
5. 文档能指导新人和 AI 代理直接开工
6. 回归错误更容易被 harness tests 提前挡住
7. 发布前需要“拍脑袋确认”的事越来越少

这些指标起来了，才说明仓库真的进入了可持续开发状态。

---

## 14. 哪些信号说明 Harness 又在变烂

反过来，如果出现下面这些信号，就要警觉：

1. 新需求总是先写代码，后补解释
2. 不同服务都在偷读同一类运行时字段
3. graph、page、gateway 都在做私有兼容兜底
4. 文档越来越像历史遗迹，没人看、也不能指导实现
5. 测试只能证明“没报错”，不能证明“符合范式”
6. 排障必须依赖某一个老手的记忆
7. 每次交付都要人工口头交接一大堆隐性信息

这几个一旦冒头，就说明 Harness 需要立刻修，不然后面只会越来越疼。

---

## 15. 推荐的下一步建设动作

对当前仓库，建议按下面顺序继续补齐 Operating Model：

### 15.1 先补模板

优先补这几类模板：

- 变更模板
- 验收模板
- 故障复盘模板
- runbook 模板

### 15.2 再补门禁

- Runtime contract harness tests
- 静态 graph harness tests
- tool registry harness tests
- 最小链路 smoke 检查

### 15.3 最后再补更重的自动化

- CI 结构检查
- 自动 smoke
- 自动回归摘要
- 发布前检查清单脚本化

别一上来就搞一堆重自动化，规则和模板没钉住之前，那些东西只会越搞越乱。

---

## 16. 一句话总结

AITestLab 的 Harness Operating Model，本质上就是一句话：

> 不让需求直接撞代码，而是要求任何改动都必须经过层级判定、契约澄清、设计落点、实现、验证、文档补齐和交付闭环。

只有这样，Harness 才不是挂在墙上的理念，而是能每天指导团队和 AI 代理稳定干活的工作方法。
